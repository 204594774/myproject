# ***\**\*\*\*摘要\*\*\*\*\****

针对大学生创新创业项目“申报材料分散、审批链条长、过程跟踪难、评审统计慢”等问题，本文设计并实现了大学生创新创业项目管理系统。系统采用 Flask + Vue3 + SQLite 技术方案，面向学生、指导老师、审批人员、评审老师、项目管理员、系统管理员六类角色，提供项目申报、过程提交、多级审批、评审打分、结果发布、通知提醒和统计分析等功能。系统按照任务书要求强调“操作简单、界面简洁、具备实用性”，并在实现中统一了角色权限、状态流转和数据留痕。测试结果表明，系统可满足学院日常项目管理需要，能够明显减少人工汇总工作量，提高流程透明度与处理效率。

关键词：创新创业；项目管理；审批评审；信息系统

# ***\**\*\*\*Abstract\*\*\*\*\****

To address common problems in college innovation and entrepreneurship projects, such as scattered application materials, long approval chains, difficult process tracking, and low review efficiency, this thesis designs and implements a project management system. The system is built with Flask, Vue3, and SQLite, and serves six roles: students, advisors, approvers, reviewers, project administrators, and system administrators. It supports project application, process submission, multi-level approval, scoring and review, result publishing, notifications, and statistics. Following the task requirements, the system focuses on simple operation, clear interface, and practical usefulness. Test results show that the system can effectively support daily project management in colleges and improve both transparency and efficiency.

Key words: Innovation and Entrepreneurship; Project Management; Approval and Review; Information System

# ***\**\*\*\*引言\*\*\*\*\****

大学生创新创业项目是高校实践教学的重要组成部分，项目数量逐年增加，管理流程也越来越复杂。传统管理方式以表格和线下沟通为主，容易出现信息重复录入、审批进度不透明、评审数据难统计等问题。相关研究指出，高校项目管理平台应覆盖“申报—审批—过程—评审—归档”完整链路，并突出角色协同与过程留痕<sup>[1]</sup>。

系统应完成需求分析、技术实现与测试，且满足操作简单、界面简洁、实用性较强等要求。基于该要求，本文将系统目标确定为：
（1）统一六类角色的业务入口；
（2）打通申报、审批、评审和结果发布流程；
（3）实现过程可查询、结果可统计、操作可追踪；
（4）控制实现难度，保证可讲清、可演示、可维护。

# ***\**\*\*\*1 开发工具及技术说明\*\*\*\*\****

## ***\**\*\*\*1.1 开发环境\*\*\*\*\****

系统开发与运行环境如下：Python 3.x、Flask、Vue3、Element Plus、Axios、SQLite。开发操作系统为 Windows。接口调试使用浏览器与 Swagger 页面。

## ***\**\*\*\*1.2 后端技术\*\*\*\*\****

后端采用 Flask 框架，按认证、项目、评审、用户、系统管理等模块拆分，整体实现结构清晰、部署简单，适合毕业设计规模<sup>[2]</sup>。数据持久化采用 SQLite，以满足本课题数据规模与本地化部署需求。后端统一返回成功或失败结构，方便前端统一提示。

## ***\**\*\*\*1.3 前端技术\*\*\*\*\****

前端采用 Vue3 + Element Plus。该方案开发效率高，组件丰富，适合管理系统页面快速搭建<sup>[3]</sup>。系统页面以表格、表单和状态标签为主，重点保证“看得懂、点得快、反馈明确”。

## ***\**\*\*\*1.4 关键实现原则\*\*\*\*\****

为保证系统可用且便于答辩说明，实现中遵循四条原则：
（1）先流程后细节：先打通业务闭环，再补充优化；
（2）先规则后页面：先定角色权限与状态流转，再做交互；
（3）先可用后高级：优先满足任务书实用性要求；
（4）先留痕后统计：关键操作必须可追踪。

# ***\**\*\*\*2 需求分析\*\*\*\*\****

本系统主要面向高校大学生创新创业项目的申报、过程管理、审批与评审业务场景。结合高校项目治理常见组织结构，系统用户划分为六类：学生用户、指导老师、审批人员、评审老师、项目管理员、系统管理员。不同角色的关注点和权限边界不同，需要在需求层面做分角色分析，并在后续模块设计中一一对应<sup>[4]</sup>。

### ***\*2.1学生用户需求分析\****

学生是项目的申报主体和实施主体，其需求如下：
（1）账号与个人信息管理：使用学号或账号登录系统，维护姓名、学院、专业、联系方式等信息，并支持密码修改；
（2）项目申报与编辑：在线填写项目名称、项目类型、所属学院、项目年度等信息，维护成员与分工，上传申报书及相关附件；
（3）过程与成果提交：在线提交中期检查材料和结题材料，必要时补充阶段成果证明；
（4）状态查询与结果反馈：查看项目在学院审批、学校审批、评审、公示等环节的状态变化，查询审批意见与评审评语；
（5）通知公告与提醒：接收申报截止、补材料、结果发布等系统提醒。

### ***\*2.2 指导老师需求分析\****

指导老师负责过程指导与质量把关，其需求如下：
（1）项目查看与指导：查看本人指导项目及申报、中期、结题材料，并给出指导意见；
（2）过程把关与确认：对项目申报条件、材料完整性进行初审，给出是否同意意见；
（3）查询与统计：查询历年指导项目的立项、结题、获奖情况，支撑教学与科研考核。

### ***\*2.3 审批人员需求分析\****

审批人员包括学院审批人和学校审批人，核心需求如下：
（1）项目审批与流转：按权限范围执行通过或驳回，填写审批意见并驱动状态流转；
（2）待办与已办管理：区分待处理与已处理项目，支持条件筛选和快速定位；
（3）统计分析与报表：按学院、年度、类型统计申报与通过情况，导出审批结果。

### ***\*2.4 评审老师需求分析\****

评审老师主要承担学术与创新性评价，其需求如下：
（1）评审任务管理：查看系统分配任务，查询历史评审记录；
（2）评审过程支持：在线查阅项目材料，按评分指标完成打分与评语填写；
（3）结果提交与查询：提交评审结果，供系统汇总排序和结果发布。

### ***\*2.5 项目管理员需求分析\****

项目管理员是业务总控角色，其需求如下：
（1）项目全生命周期管理：全局查看申报、立项、中期、结题等阶段分布，处理项目变更；
（2）赛事与活动管理：维护校赛、省赛、国赛等活动信息及项目关联；
（3）评审组织与专家管理：维护专家信息，设置评分规则，分配评审任务并汇总结果；
（4）统计报表与数据分析：生成多维统计报表，支撑教学质量评估和管理决策。

### ***\*2.6 系统管理员需求分析\****

系统管理员负责平台运维与安全，其需求如下：
（1）账号与权限管理：维护用户账号，配置菜单权限，执行启停用与密码重置；
（2）基础数据与参数维护：维护学院、专业、项目类型、项目级别等基础数据和系统参数；
（3）安全与日志审计：执行备份恢复，审查运行日志与操作日志，发现异常访问；
（4）系统运行维护：支持版本升级、公告维护和用户问题处理。

# ***\**\*\*\*3 系统设计\*\*\*\*\****

# ***\**\*\*\*3.1 总体设计\*\*\*\*\****

系统采用 B/S 架构，前端负责交互展示，后端负责业务逻辑处理，数据库负责数据持久化。整体交互路径为“用户操作页面→请求后端接口→后端鉴权与处理→数据库读写→返回结果”。该架构在高校管理平台建设中应用较为广泛，具备部署简单、维护成本低、迭代效率高等特点<sup>[5]</sup>。

从工程实践角度看，系统围绕“够用、稳定、可解释”进行设计，采用轻量的单体分层结构。核心能力分为表示层、业务层、数据层和支撑层四层，其中支撑层主要承载鉴权、日志、统一返回和参数配置等共性能力。

## ***\**\*\*\*3.3 数据库设计\*\*\*\*\****

数据库围绕“用户、项目、评审、日志”四类核心数据设计。主要表包括用户表、项目表、项目成员表、评审任务表、评审结果表、通知表、系统日志表。

设计时重点考虑三点：
（1）状态字段清楚，便于流程推进；
（2）关联关系简单，便于查询统计；
（3）关键时间和操作者可追溯，便于审计。
数据库设计思路参考了常见教学管理系统的数据建模方式<sup>[6]</sup>。

在结构设计上，项目表是主业务表，承担项目基础信息、当前状态、审批结果和发布时间等字段；项目成员表通过外键关联项目表，支持一对多成员管理；评审任务表与评分结果表用于实现“任务分发—评分提交—结果汇总”的评审闭环；系统日志表记录关键行为，支撑问题追溯和过程审计。

为了保证后续统计效率，系统在设计时对高频查询字段进行规范化，如项目年度、学院、项目类型、评审阶段、状态字段等，避免把统计全部压在非结构化文本字段上。该策略在实际测试中能够明显减少管理端列表页与统计页的查询成本<sup>[7]</sup>。

## ***\**\*\*\*3.4 业务流程设计\*\*\*\*\****

## ***\**\*\*\*3.6 可维护性与可扩展性设计\*\*\*\*\****

系统不仅要“能运行”，还要“可持续维护”。因此在设计阶段引入了可维护性与可扩展性要求，并在实现中落实为具体策略。

第一，模块边界清晰。认证、项目、评审、系统管理分别独立，避免业务逻辑交叉污染。当某一模块需要迭代时，开发者可以在不影响其他模块的前提下进行局部调整。

第二，配置优先于硬编码。流程节点、角色权限、统计筛选项尽量采用配置化管理，减少业务规则变动时的改代码成本。这种做法符合软件工程“高内聚、低耦合”原则<sup>[8]</sup>。

第三，数据结构标准化。数据库字段命名统一、状态值含义明确，避免同义字段和冗余字段并存问题。该策略能显著提升查询和统计的可维护性，也符合数据库规范化设计思想<sup>[9]</sup>。

第四，日志与错误码体系化。对常见异常进行分类返回，便于前端提示和后端排查，减少“问题发生了但难以定位”的情况。

为了便于后续扩展，系统预留了三类扩展点：
（1）流程扩展点：可增加新的审批节点或评审阶段；
（2）统计扩展点：可增加按专业、导师、比赛类型的统计维度；
（3）终端扩展点：可在保持后端接口不变的前提下接入移动端或小程序端。

以上设计使系统在课程设计阶段即可具备较好的长期演进基础。

# ***\**\*\*\*4 系统详细实现\*\*\*\*\****

## ***\**\*\*\*4.1 学生用户模块实现\*\*\*\*\****

学生端实现内容包括：登录与信息维护、项目申报、中期材料提交、结题材料提交、状态查询。学生提交项目后，系统会记录提交时间和当前状态，并进入审批环节。学生页面的功能组织与图3-1中的学生模块保持一致。

学生模块实现重点是“少填错、易修改、能追踪”。因此系统对必填字段和附件进行前置校验，并支持被驳回后修改重提。

在交互细节上，学生端采用“分块式表单+状态标签提示”的方式，并对项目名称、项目类型、负责人信息、成员列表、附件清单等关键字段进行必填校验；缺失信息时页面直接提示缺失项，减少提交后反复回退修改<sup>[10]</sup>。状态查询同时展示当前状态与历史处理意见，便于学生理解原因与下一步动作。

## ***\**\*\*\*4.2 指导老师模块实现\*\*\*\*\****

指导老师模块实现了项目查看、意见填写和过程确认。老师可查看本人指导项目，给出“同意/修改后再提”等意见。中期和结题阶段也可进行意见反馈。

该模块直接对应图3-1中的“项目查看与指导、过程把关与确认”两个子功能。

为了保证教学指导有效性，系统在老师侧采用“意见结构化记录”，支持结论与修改建议同时留存，便于学生按项修改并支撑后续统计。

同时，老师仅能查看与本人相关项目，系统在接口层根据用户身份和项目关系进行过滤，避免无关项目暴露，满足最小授权原则<sup>[11]</sup>。

## ***\**\*\*\*4.3 审批人员模块实现\*\*\*\*\****

审批模块实现了学院和学校两级审核流程，支持待办列表、已办列表、驳回理由记录和状态自动流转。审批通过后项目进入下一环节，驳回后返回学生修改。

审批权限边界按表3-1执行，防止跨权限误操作。

从实现机制上看，审批模块的核心是“状态迁移与意见留痕同步发生”。每次审批动作都会同时写入新状态、处理意见、处理人和处理时间，确保后续追溯链完整。若仅更新状态而不记录意见，会导致管理上出现“状态可见、原因不可见”的信息断层。

在效率方面，审批端提供待办与已办分区，并支持按学院、年度、项目类型筛选，降低管理人员在高峰期处理大量项目时的查找成本。该做法在高校流程审批系统中较为常见<sup>[12]</sup>。

## ***\**\*\*\*4.4 评审老师模块实现\*\*\*\*\****

评审模块实现了任务接收、在线打分、意见提交和历史记录查询。评审老师只可看到系统分配给自己的任务，提交后不可随意修改。

评分结果由系统汇总，用于排名与结果发布。该模块对应图3-1中的“评审任务管理、评审结果提交与查询”。
该设计与项目评审系统“任务分配+统一汇总”的主流方案一致<sup>[13]</sup>。

为提高评审公平性，系统在评审页面中对项目材料、评分项和意见输入进行分区布局，尽量减少误操作。评分提交前系统进行范围检查，避免出现超出评分区间或未填写必填评价的情况。

评审任务处理完成后，系统保存评分时间与评审人信息，形成完整审计链。该链路不仅用于结果汇总，也可在出现争议时回溯过程，保证评审透明和可解释性。

## ***\**\*\*\*4.5 项目管理员模块实现\*\*\*\*\****

项目管理员模块实现了全局项目列表管理、评审任务分配、结果发布和统计分析。管理员可按学院、类别、年度筛选项目，并查看通过率、结题率等统计结果。

该模块与图3-1中的“项目全生命周期管理、赛事活动管理、统计分析”保持一致。

项目管理员模块是系统业务中台。其关键实现点包括三类：
（1）评审组织能力：为不同项目分配评审老师并监控任务完成率；
（2）发布治理能力：对结果发布进行完整性检查，避免缺失数据直接公示；
（3）统计决策能力：按学院、项目类型、年度输出统计维度。

在实践中，管理员最关注的是“当前阶段还有多少未处理”“哪类项目通过率偏低”“哪些学院提交延迟”。因此系统统计页采用面向管理问题的指标组织方式，而不是单纯罗列数据字段。

## ***\**\*\*\*4.6 系统管理员模块实现\*\*\*\*\****

系统管理员模块实现了账号与角色维护、基础数据维护、日志审计、备份恢复等功能。该模块用于保障平台稳定运行。

权限和功能范围按照表3-1执行。后续章节再次提到权限划分时，只写“如表3-1所示”。

系统管理员模块还承担系统健康维护职责。通过查看运行日志、接口异常日志和关键操作日志，管理员能够快速定位故障点。对于课程设计系统而言，虽然部署规模不大，但日志体系仍然是保障系统可维护性的基础能力。

在备份恢复方面，系统支持数据库备份，能够在误操作或异常情况下进行恢复，降低数据损失风险。该做法符合高校信息系统“可恢复、可追踪、可审计”的基本要求<sup>[14]</sup>。

## ***\**\*\*\*5.1 测试方法\*\*\*\*\****

测试采用功能测试与流程测试相结合的方法，按六类角色分别执行关键操作，并进行跨角色流程联调；覆盖登录权限、申报提交、审批流转、评审打分、结果发布、日志记录等关键场景<sup>[15]</sup>。测试环境与开发环境一致，测试记录包含输入条件、操作步骤、预期结果、实际结果与是否通过。

（3）评审结束后，项目管理员可发布结果并查看统计；
（4）系统管理员可在日志中追踪关键操作。

整体看，系统功能符合任务书“操作简单、界面简洁、具备实用性”的要求。

## ***\**\*\*\*5.3 不足与改进\*\*\*\*\****

当前系统仍有改进空间：
（1）统计页面可进一步优化加载速度；
（2）移动端适配还不完善；
（3）部分提示文案可进一步简化。

这些改进不影响当前版本完成课程设计目标。

进一步看，后续改进可按“先稳定、后优化、再扩展”推进：
（1）稳定层面：继续完善异常提示和边界校验，减少误操作；
（2）优化层面：优化统计页面查询策略，提升数据加载速度；
（3）扩展层面：增强移动端适配与图表分析能力，提升使用体验。

该改进路径与高校管理类系统迭代实践一致，即优先保证流程可靠，再逐步提升交互与性能。已有相关研究也表明，流程可视化、组件化前端、权限模型优化、统计分析增强和日志审计完善是同类平台演进的主要方向<sup>[17]</sup>。

## ***\**\*\*\*5.4 测试结论的量化分析\*\*\*\*\****

为增强测试结论的客观性，围绕流程完整性、权限准确性和可追溯性三方面进行验证：主链路多轮回归未出现状态回跳与终态缺失；跨角色越权访问均被拒绝；关键操作均可在日志中追溯到操作者、时间戳与目标对象。异常输入可返回明确提示，综合表明系统达到预期目标。

本文完成了大学生创新创业项目管理系统的需求分析、系统设计、功能实现与测试验证。系统围绕六类角色构建完整业务闭环，实现了申报、审批、评审、发布和管理等核心功能。
